# 19.2    Object-Oriented Design
オブジェクト指向でソフトウェアプロジェクトを設計するのは難しい、いや、非常に難しい。技術的な手法については本書の他の箇所で詳しく解説している一方、設計手法については、本棚数メートルを埋めるに相当するほど書物が存在するのだが、本書では、自分が設計に役立ててきたいくつかのヒントを紹介するに留める

抽象化を行い、クラスを設計し、カプセル化を施しているときはいつも「今、私がやっているのはオブジェクト指向か？」と自問するだろう。100%「そうだ」と答えられることはまずない。それでも、抽象化は現在の問題をきれいに解決するための必要な範囲のみに留めるべきである。

自分が正しい方向に進んでいるかどうかを確認する糸口がある。single, open, Liskov, interface, dependency（SOLID）原則に基づいて自分の設計を繰り返し検証したり、SRP、OCP、LSP、ISP、DIPといった略語を使ったりすることが役立つ。

## 19.2.1    SOLID

- Single responsibility principle
  - クラスは一つの responsibility のみを持つべきであり、それゆえ変更されるべき理由も一つだけであるべき
    - この有名なルールは、ロバート・C・マーティン（「アンクル・ボブ」）によって最初に提唱された
    - このルールは、クラスのソースコードを変更すべきタイミングを規定している
    - 彼は、ソースコード変更の理由を、クラスの責任範囲内の変更と定義し、クラスは一つの責任のみを持つべきであると述べている
  　- 彼の例として、内容と表示形式が変更されるレポートが挙げられます
    - これらは二つの責任であり、したがって二つのクラスによって処理されるべき
    - このルールの具体的な実装例は、モデル・ビュー・コントローラー（MVC）の概念、あるいはより新しいモデル・ビュー・ビューモデル（MVVM）のアプローチに見ることができる

- Open/closed principle
  - クラス、関数、モジュールは拡張に対しては開いているべきだが、変更に対しては閉じているべき
    - 例えば、自身のソースコードを変更することなく機能を拡張できるようなクラス
      - 抽象インターフェースの適用はこの原則に由来するもので、純粋なインターフェース（実際には関数宣言の集合）を定義し、それを異なるクラスで複数回実装する
        - これは、継承、動的ポリモーフィズム、仮想関数によって効果的に実現できる
        - C++では思うままに使える動的な手段は少なく、関数オーバーロード、テンプレート特殊化、type trait、動的ライブラリなどである。
          - Javaから来た人にとっては、この動的アプローチの少なさに馴染みがないかもしれないが、少なくとも同等の力はある

- Liskov substitution principle
  - VwBus（フォルクスワーゲンのバン）がCarのサブタイプである場合、Car型のオブジェクトをCar型のプロパティを変更することなく、VwBus型のインスタンスに置き換えることができる
    - なぜなら、車はなにができないといけないだろうか？運転とクラクションだ
    - 派生オブジェクトはキャンプすることもできるが、車にはそれは必要ではない
    - したがって、車が必要な場所ならどこでもVwBusを使用できる
  - C++の場合、これはより実践的に表現できる
    - 基底クラスへのポインタまたは参照を使用する関数は、意識することなく派生クラスも処理できなければならない

- Interface segregation principle
  - きれいなインターフェースの分離はユーザーが不要なメソッドに依存しない事を保証する
  - メソッドが多数含まれるインターフェースを避け、より小さく具体的なインターフェースに分割することで、ユーザーは実際に必要なメソッドだけを知ればよくなる
  - もし必要になることがあれば -例えばリファクタリングの一環として- 後から大規模なクラスを異なるものに抽出、分割、再配置することができ、その変更に気づく呼び出し元はごくわずか
  - 適切に設計されたインターフェースは、システム間の結合度が低いため、テストの容易性にも貢献します。

- Dependency inversion principle
  - 目標は、上位のモジュールを下位のモジュールに依存させないこと
    - 両者を分離する抽象レイヤーを両者で使用することで達成できる
    - ここでもMVCは良い例で、データと密接に結合したGUIを構築する代わりに、（データ）モデルとビューを分離し、その間に作成した多くの抽象クラスを持つコントローラーで両者を接続する

### Model-View-Controller
この広く普及しているアーキテクチャパターンは、プログラムを3つのコンポーネントに分割し、「すべてが互いに通信する」よりも制御された方法で通信させます。コントローラはビ​​ュー（プレゼンテーション）とモデル（データモデル）を制御します。プレゼンテーションはデータの格納方法とは独立している必要があり、データの格納方法は使用されるビューとは一切関係ありません。

例えば、データストレージをOracleデータベースに実装し、ウェブサイトの表示をHTMLで実装したとします。コントローラーがこれら両方を統合し、すべてが正常に動作します。
